Bring the app back up.

The /account 500 (and the whole app) traced to a query in the shared layout that crashed every page. The crash is fixed in code — here's exactly how to ship it and clear the root cause.

What broke

The shared (app) layout loads your dashboard data on every page. That query threw on an error, so one bad query 500'd the entire app — account, dashboard, all of it. A likely DB column was missing because a migration hadn't been applied. The fix makes that query log the cause and return empty instead of crashing.

Do this, in order

Three steps to green.

  1. Apply the missing migration in Supabase

    Open Supabase → SQL Editor → New query, paste the file, click Run. You've confirmed 0006–0008; the usual missing one is 0005 (it adds requests.shipping_address, the addresses table, and avatars).

    ✓0005avatars · addresses · shipping_address
    ✓0006notification prefs
    ✓0007profile email mirror
    ✓0008request estimate

    All four must show as applied. If unsure, just re-run each in order — they use add column if not exists, so re-running is safe.

  2. Ship the code fix to production

    The fix is on branch feat/launch-prep (commit c63ebea). Merge it to main so production rebuilds with it.

  3. Open the PR: github.com/chakra3301/FindersXKeepers/pull/1
  4. Click Merge pull request → Vercel auto-deploys main to production.
  5. Wait for the deployment to go Ready (and confirm it shows commit c63ebea or later).

  • Reload and verify

    Hard-refresh findersxkeepers.com/account. It should load now. Then click into the dashboard and add a profile picture to confirm writes work.

    If a page loads but looks empty, the real query error is now in your logs — see below.

    • If anything still looks off

    Read the cause in Vercel logs.

    1. Open the runtime logs

      Vercel → your project → the Production deployment → Logs tab (a.k.a. Runtime Logs / Functions).

    2. Trigger it, then read the line

      Reload the broken page to generate a log entry, then look for one of these — it names the exact failing table/column:

      # examples you might see
[getDashboardRequests] 42703 column requests.shipping_address does not exist
[getProfile] 42P01 relation "profiles" ...
[listAddresses] could not find the table 'public.addresses'

      A does not exist / could not find the table message = a migration still isn't applied. Run it (step 1) and you're done.

    After this, the app is hardened: the layout query, profile/address loads, and the estimator/email config all degrade gracefully + log instead of crashing — so a missing migration or a mis-pasted env var can never take the whole site down again.
